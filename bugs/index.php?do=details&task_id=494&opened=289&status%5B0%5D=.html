Hi,

I first noticed this on an older shifty version, no idea if it can happen with newer versions too.

I wrote this to awesome's rc.lua (yes, this is the whole content and no, it is not usable besides as a test case):

require("awful")
if true then
my_tag = tag(1)
my_tag.screen = 1
my_tag.selected = true
end
awful.hooks.manage.register(function (c, startup)
if my_tag then
awful.client.movetotag(my_tag, c)
end
c.screen = 1
client.focus = c
end)

For seeing this bug in action, start awesome and an xterm (anything will do). Then I kill awesome, change the second line in rc.lua from "if true then" into "if false then" and start awesome again. The xterm disappears.
I think awesome should somehow make sure this doesn't happen...

Comment by koniu (koniu) - Thursday, 09 April 2009, 14:56 GMT
This is more of a feature than a bug. It's the lua side that's responsible for creating tags and, to certain extent, putting clients in them. If your simplified rc.lua doesn't create any tags then there won't be any to display the xterm in.

The default rc.lua, in this respect, creates a number of tags on load and then (for tagging new clients) relies on at least one tag being selected. Note that if you run the default config then deselect the tag so that none are selected (right click on taglist) and run xterm, you won't see it either.

Shifty instead of creating the tags on load, waits for a client to open (manage hook, shifty.match()) and then creates a respective tag depending on the configuration and moves the client there.

I would actually go one step further and move the 'withcurrent' functionality from a separate hook in awful.tag to rc.lua manage hook (where we already handle focus eg). This would give people more liberty in client<->tag manipulation. Example: in shifty new clients are needlessly tagged with selected tags and then have to be moved to their preconfigured tags instead of being tagged with them in the 1st place.
