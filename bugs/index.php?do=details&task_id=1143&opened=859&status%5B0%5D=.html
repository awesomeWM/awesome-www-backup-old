Details

NetworkManager (v0.9.6.4) applet icon in systray is flickering.

If nm-applet is executed with NMA_SIZE_DEBUG=1 env var, one can see endless cycle of

** Message: status_icon_size_changed_cb(): status icon size now 19
** Message: status_icon_size_changed_cb(): status icon size now 22
** Message: status_icon_size_changed_cb(): status icon size now 26
** Message: status_icon_size_changed_cb(): status icon size now 28

debug output, which is printed in handler of "size-changed" signal of gtk3 gtk_status_icon_* object. So it apparently is some kind of race condition. Since I am not expert in X windows internals, I cannot say for sure if it is a fault of awesome, or rather of nm-applet, however this behavior does not occur if awesome's own systray is disabled and instead "tint2" is used for systray.

Note that size is 28x28 and not 19x19 as for the rest of tray icons (I left just one).

Comment by Uli Schlachter (psychon) - Tuesday, 11 June 2013, 13:53 GMT
The problem seems to be that NM wants it tray icon to have a size of at least 28x28 pixels:

WM_NORMAL_HINTS(WM_SIZE_HINTS):
program specified minimum size: 28 by 28
program specified base size: 0 by 0
window gravity: NorthWest

However, your systray seems to have only space for 19 pixels in height (0x100000b (has no name): () 1024x19+0+0 +0+0)
(Also, awesome ignores WM_NORMAL_HINTS on the systray anyway)

However, NM insists on its icon size and the following happens (I picked out a random time range of the xtrace output (I chose 6.179secs/seq 0385 to 8.166secs/seq 047a):

At 6.179: NM resizes its tray icon to 28x28 pixels.
At 7.218: The icon is resized to size 19x19 (likely awesome just happened to redraw the wibox and thus the systray icon sizes were enforced again)
NM reacts by redrawing its icon (and querying the mouse pointer state...?)
At 7.227: NM resizes its icon to size 22x22 (Hu? This is not 28x28)
At 7.235: NM resizes its icon to size 26x26 (Still smaller than expected)
At 7.244: NM resizes its icon to size 28x28 (Finally there!)
Of course, after each resize, the icon is redrawn and the mouse pointer state is queried.
At 8.142: Awesome resizes the tray icon to size 19x19.
As above, NM reacts by sending lots of pointless traffic until it finally settles at size 28x28 at time 8.166. A second later, awesome kicks in again.

Do you happen to have a textclock showing seconds and thus your wibox is redrawn once per second?

For the issue at hand:
Ok, awesome is ignoring the tray icons minimum size. However, I don't see what the alternative is (there just is no space for an 28x28 icon) and NM is really stupid.
Why does it have to redraw its icon 5 times and resize it three times just because awesome resized the window? Each of those causes some flickering. IMHO NM would best give up and go with the 19x19 size that it was given.

I could make awesome enforce the 19x19 size so that NM can no longer resize its icon. However, I think that is a stupid thing to do and it feels like it just works around NM's behavior...
