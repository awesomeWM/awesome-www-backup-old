Details

Using latest awesome-git checkout, but this is an ongoing problem as of some months now. (Arch Linux: `yaourt -Sa awesome-git`).

In java applications, when running a dual-head (xinerama) setup, there are a variety of strange behaviours with menus in java applications. (Tested with, amongst others, CGoban3 from http://www.gokgs.com). This occurs in both Oracle's latest JRE and OpenJDK 7.

On the first/leftmost monitor, a menu often responds to a click and stays open, but is 'offset' by perhaps 20 vertical pixels. (So I can select an item by hovering in the space above it.)

On the second/rightmost monitor, the menu will not stay open, but if I hold down the mouse and move to the equivalent space on the first/leftmost monitor it starts to select items.

In a single-monitor setup Java seems to work fine even without the _JAVA_AWT_WM_NONREPARENTING and wmname hacks. The behaviour described here occurs regardless of whether I'm using _JAVA_AWT_WM_NONREPARENTING/wmname or not. This clearly seems related to the long-running reparenting issue. I thought that wmname, and the OpenJDK, had made this a non-issue, though.

To reproduce:

1) Run awesome in a xinerama dual-monitor setup.
2) Run a java swing application, such as CGoban3 (www.gokgs.com)
3) Move window between workspaces and try clicking on menus.

I noticed in Task #972 that someone mentioned 'the java menu problem', but I couldn't find an actual bug report linked to this, so I opened one.

my fault.
the screenshots were taken with the "wmname LG3D" hack

without, the menus are on the wrong location (they are positioned absolute instead of relative to the parent window), but the mouse was correctly recognized above them. (see broken.jpg)

after some poking, i found out that moving the window fixes everything, and resizing breaks everything again:

in my case i did it with pycharm (like intelliJ or any other java application)

running pycharm in some non-floating layout: broken
set pycharm to float without moving: broken
moving the window: okay again
resizing the window: broken
moving it again: okay again

anyways:

setting send_notice = true in objects/client.c:client_resize_do

corrects this behavior, but as psychon pointed out on IRC this is not a desirable fix
