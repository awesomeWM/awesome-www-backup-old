Comment by Uli Schlachter (psychon) - Saturday, 10 November 2012, 11:35 GMT
First some observations: luadoc can be partly unbroken by removing the "--nofiles" argument, but the result is ugly. Stuff gets a little nicer via lots of magic and redefining module() via "function module() end". This way we can do module("foo") everywhere again. However, this is ugly and results in ugly documentation. So I guess staying with luadoc isn't much of an option.

So I guess this means that ldoc wins. As a first guess, this involves multiple steps:
- Change CMakeLists.txt and awesomeConfig.cmake to use ldoc instead of luadoc ("git grep -i luadoc" should find all the places)
- Making sure all the existing lua doc looks fine and doesn't do anything that ldoc doesn't support
- Improve current documentation (e.g. using markdown instead of html, dunno what else ldoc does that luadoc can't do)
- Simplify the documentation for the C api. Currently this is done by having special files in luadoc/ which look a lot like lua and can be fed to luadoc. However, ldoc can directly parse docs in C files and I think we should use this.

At least the first two points of this list should be done for awesome 3.5, the others could be postponed.

Did I miss anything on my list? Any suggestions?
Me being lazy, I'd like to ask if anyone wants to port awesome from luadoc to ldoc. This shouldn't be all too hard. I guess the most complicated part would be the changes for CMake which I can do (if someone tells me with which arguments to call ldoc ;-) ).
