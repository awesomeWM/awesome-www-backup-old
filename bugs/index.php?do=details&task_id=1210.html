Details

I recently noticed that true transparency is working in Awesome again. I remember it working in 3.4 a couple of years ago, but after having updated to the git version it stopped working. I made a bug report and it was fixed by the adding of pseudo transparency to wibox. A couple of months ago (for some reason I didn't report it back then) I noticed that true transparency started working again after removing the pseudo transparency hack from wibox/drawable.lua. That means transparent wiboxes, titlebars, etc.

I have no idea if many people rely on the pseudo transparency hack, but to get true transparency working all you need is to use a compositor (xcompmgr, compton, etc).

Should true transparency be implemented? The attached patch removes pseudo transparency from wibox.

drawable: Support for true transparency ( FS#1210  )

This makes the drawable paint the wallpaper in the background
(i.e. pseudo transparency) only if there is not a composite manager running,
as reported by the value of awesome.composite_manager_running.

In other words: drawables can now make use of true transparency.

For obvious reasons (just look at it!), the patch won't be accepted as-is (and for less obvious reasons: I think that after removing the wallpaper-changed signal, there is no more reason for the drawables table to exist and it could be removed, too...).

However, the idea still doesn't work. Try killing your compositing manager and you will get no transparency and instead the transparent parts of the wibox become black. Semi-transparent parts become semi-black. Obviously this isn't acceptable.

I guess that this should be fixed by checking for the _NET_WM_C-something property / selection that compositing managers are supposed to own (= check if a compositing is running) and only then use a true-transparent background for the wibox.

(Also, I just noticed what an ugly shitload of crap I wrote. Awesome retrieves the wallpaper from the X server each time a wibox is reparented...)
