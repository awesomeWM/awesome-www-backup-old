Details

Sometimes when I'm too far away from the keyboard, I use my wireless mouse, and the on-screen keyboard, kvkbd. This arrangement used to work perfectly for me until a particular commit: 03759b48478eb716af8b62e01ef3b5140c582f58, titled "Fix keyboard layouts", and dated "Feb 1 23:06:08 2013 +0100"

Before this particular commit, I was able to use all control keys like Win, Ctrl, Alt, Shift, etc. to issue commands to awesome and to other programs. After this commit, the control keys seem to work fine for other programs, but not for awesome itself.

So, pressing Win-R on kvkbd no longer has any effect.

If you must have this change, XkbIgnoreExtension(True), can you please allow a way to control this behaviour via rc.lua?

Longer explanation:
Yes, it did work with awesome 3.4.12 and earlier. There were some limitations up to 3.4.12: the Win key would not work. For example, to launch dmenu (my own keybindings), I'd have to use Ctrl+Alt+R instead of Win+R. I then set up my hotkeys to respond to either combination, so I could still live with it. With the original 3.5 release, even the Win+key combos started working perfectly. The only combos that never worked were the Win+Mouse1 and Win+Mouse2 for moving and resizing -- which is perfect, because if it did, I'd be unable to click on any other button in kvkbd, as soon as I pressed Win. This particular git patch broke all that and more, and I was unable to use Ctrl and Alt either, like I used to in 3.4.12.
