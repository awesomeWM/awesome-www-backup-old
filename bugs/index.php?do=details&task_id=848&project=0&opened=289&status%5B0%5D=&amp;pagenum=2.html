Details

It seems like other WMs handle _NET_ACTIVE_WINDOW messages for windows that aren't visible by switching to that window's workspace and then activating the window. Awesome handles this by ignoring the message (which is fine according to the wording of EWMH).

It would be interesting if someone could hack a small app that opens some gtk window and calls gtk_window_present() on it with a timer (say, every 15 secs), so that we can compare awesome's behavior with other WMs.


Look at the following mailing list thread for details:

http://news.gmane.org/find-root.php?message_id=%3c20101108084252.GA30477%40upsilon.cc%3e
(This mail handles the behavior of gtk_window_present(): http://article.gmane.org/gmane.comp.window-managers.awesome/6479 )

client: Add request::activate signal ( FS#848 )

When we receive a _NET_ACTIVE_WINDOW request on the root window, we used to just
focus and raise the window. This didn't do much for clients which are on
non-selected tags.

Thus, this commit makes awesome emit request::activate on the client instead.
This is used in awful.ewmh to implement the old behavior again, but with
additionally marking the client as urgent if it isn't visible.

People who don't like this behavior can use client.disconnect_signal to disable
this behavior again. To make this really possible, awful.ewmh becomes a
"non-nil" module.
